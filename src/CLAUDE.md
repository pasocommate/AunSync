# C++ コメント記述ルール

## 言語ポリシー

**日本語を基本**とする。固有名詞（OBS, FFmpeg）・型名・変数名はアルファベット表記。

---

## Docコメントの使い分け

### `/** */` — クラス・構造体の概要説明（複数行）

```cpp
/**
 * RTSP ストリームを FFmpeg サブプロセスで受信し、
 * 注入プローブ信号の到達時刻から E2E 遅延を算出する。
 */
class RtspE2eProber { ... };
```

**使用場所**: public ヘッダー上の `class` / `struct` 定義

### `///<` — 1行で収まる宣言（末尾コメント）

```cpp
// メンバ変数
struct RtspE2eResult {
    bool        valid;          ///< 計測成功フラグ
    double      latency_ms;     ///< E2E 片道レイテンシ推定値（中央値, ms）
    double      min_latency_ms; ///< 試行結果の最小値 (ms)
    double      max_latency_ms; ///< 試行結果の最大値 (ms)
    std::string error_msg;      ///< 失敗理由
};

// 短い関数宣言が連続する箇所
void start();    ///< 計測ワーカーを起動する
void cancel();   ///< 実行中の計測をキャンセルする
bool is_running() const; ///< 計測ワーカーが実行中か返す
```

**使用場所**: 1行で収まる宣言すべて（メンバ変数・関数宣言）  
private メンバは非自明な場合のみ（単位・上限・特殊な初期値など）

### `///` — `@param` / `@return` が必要な関数（前置コメント）

```cpp
/// RTSP E2E 計測を開始する。計測中の再呼び出しは無視される。
/// @param rtsp_url         計測対象の RTSP URL
/// @param ffmpeg_path_hint ffmpeg.exe の配置ヒントパス
bool start(const std::string &rtsp_url, const std::string &ffmpeg_path_hint);
```

**使用場所**: パラメータや戻り値の説明が必要な public 関数  
説明が不要なら `///<` 末尾コメントを使う。

### 宣言整形ルール

1. `///<` を使う宣言は、原則として空行を挟まず連続配置する。
2. 宣言行が長くなる場合は `///<` を使わず、直前行に `///` を置く。
3. `///` + 宣言を1セットとして扱い、セット同士は1行空ける。
4. 長い関数宣言は引数を改行して折り返し、宣言行自体を短くする。
5. 桁揃え目的の過剰なスペースは使わない（横幅を増やさない）。
6. 行長の目安は100文字前後。110文字超はコメント前置き＋折り返しを優先する。

## 通常コメント `//` の使い方

### セクション区切り

```cpp
// ============================================================
// 音声処理
// ============================================================
```

60文字固定の `====` 区切り。ファイル内トップレベルのセクションに使用。

### 処理説明・注記

「**なぜそうしているか**」「非自明な制約・トリック」を記述する。  
コードを読めばわかる「何をしているか」は原則不要。

```cpp
// 遅延サンプル数 = delay_ms * sample_rate / 1000
// ロールオーバー対策: write_pos_ はモジュロを取らず加算のみ
int pos = (write_pos_ - delay_samples_ + buf_size_) % buf_size_; // 負数対策
```

## ファイルヘッダー

`.hpp` にクラス・構造体の `/** */` がある場合、ファイルヘッダーは不要。

`.cpp` は以下の基準で判断する：

| ケース | 方針 |
|--------|------|
| 設計上の制約・アルゴリズム・プラットフォーム限定など、コードを読むだけでは分からない情報がある | `/* */` で冒頭に記述 |
| それ以外（ファイル名と内容が対応している） | 不要 |

```cpp
/*
 * rtsp-e2e-prober.cpp
 *
 * 計測ステップ:
 *   1. ffmpeg サブプロセスで RTSP ストリームを受信開始
 *   2. プローブ信号（チャープ）を audio-processor から注入
 *   3. ffmpeg の受信音声からプローブ到達時刻を検出
 *   4. 注入時刻との差分を E2E 遅延として返す
 */
```

中身のないヘッダー（プロジェクト名・"Clean rewrite" など）は記述しない。

## 名前空間

| ケース | 方針 |
|--------|------|
| `ods::core` / `ods::network` 等のサブ名前空間 | そのサブシステムの**代表ヘッダー1箇所**に `/** */` を記述。他ファイルでは繰り返さない |
| `ods::ui::channels` 等の3階層 | 同上（階層が深くても代表ヘッダー1箇所のみ） |
| `namespace {}` anonymous | 原則不要。意図的にまとめたグループがある場合は直前に `// --- ファイルローカル ---` 程度でよい |

代表ヘッダーの選び方：名前空間内で最も広く include される、または定数・型定義を集めたヘッダーを優先する。

```cpp
// constants.hpp（ods::core の代表ヘッダー）
/**
 * 遅延バッファ・共通定数など、プラグイン全体で共有するコアユーティリティ。
 */
namespace ods::core {

// delay-buffer.hpp（同じ名前空間を開くが、ファイルヘッダーコメントは書かない）
namespace ods::core {
```

## `.cpp` ローカル要素のコメント

`namespace {}` 内や `static` など、外部に公開されない要素には Docコメント（`///`, `/** */`）を使わない。

| 対象 | 方針 |
|------|------|
| `namespace {}` 内のクラス・構造体 | `//` 1行で目的を説明 |
| `namespace {}` 内の関数 | 原則不要。非自明な場合のみ `//` |
| `namespace {}` 内の定数 | インライン `//` で単位・用途・由来を補足 |
| `static` ローカル変数 | インライン `//` で単位・特殊な初期値など |

```cpp
namespace {

constexpr char kStepperMagic[]     = "STEPPER\x1F"; // legacy format
constexpr char kStepperMagicPipe[] = "STEPPER|";    // current format

// OBS_TEXT_INFO プレースホルダーを StepperRow へ差し替える実体ウィジェット。
class StepperRow : public QWidget { ... };

} // namespace
```

## 書いてはいけないコメント（禁止パターン）

```cpp
// 変数を初期化する    ← コードそのまま
delay_ms_ = 0;

/// デストラクタ       ← 自明
~StreamRouter();

// i をインクリメント
i++;
```

## 既存コード適用の優先度

| 優先度 | 対象 |
|--------|------|
| 高 | public `struct` / `class` の定義 → `/** */` |
| 高 | public `struct` / `class` のメンバ変数 → `///<`（末尾） |
| 中 | public 関数（1行で収まる） → `///<`（末尾） |
| 中 | public 関数（`@param` が必要） → `///`（前置） |
| 低 | private 実装関数 → 非自明な場合のみ |
| 対象外 | getter/setter・自明な1行関数 |

## 実装スタイル（追記）

### クラスメンバの並び順

クラスメンバは、原則として `public:` → `private:` の順で配置する。
各アクセス指定子の中では、以下の順序を守る。

1. ネスト型（`using` / `enum` / ネスト `struct` / ネスト `class`）
2. フィールド類（定数・メンバ変数）
3. `inline` メソッド（クラス内定義）
4. ctor / dtor
5. その他メソッド

### 自由関数の整理

自由関数は、用途カテゴリが明確な `namespace` で整理する。
`namespace {}` に無秩序に並べず、`audio` / `network` / `ui` など意図が伝わる単位で分ける。

### 生ポインタ・生配列・生文字列の扱い

理由がない限り、生ポインタ・生配列・生文字列は使用しない。
以下のような適切なラッパーを優先する。

- 所有権: `std::unique_ptr`, `std::shared_ptr`
- 配列・バッファ: `std::array`, `std::vector`, `std::span`
- 文字列: `std::string`, `std::string_view`

固定長バッファの手作業確保、`new` / `delete` の明示操作は原則避ける。
（API 制約などで必要な場合は、理由をコメントで明示する。）
