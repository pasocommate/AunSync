# ローカル生演奏（先行チャンネル）対応

微調整タブの「ローカル生演奏を考慮」機能に関する目的・設計・実装意図・注意点をまとめる。
ディレイ計算式そのものは [time-data-structures-uml.md](time-data-structures-uml.md#遅延計算式) を参照。

## 用語

本ドキュメントは [CLAUDE.md](../CLAUDE.md) の用語規約に従う。

- **遅延 (Latency)** — 計測・想定されるシステムの遅れ（OBS配信遅延 R、想定アバター遅延 A など）
- **ディレイ (Delay)** — タイミング調整のため本プラグインが意図的に付加する待機時間

主要記号: `R` = OBS配信遅延（RTSP E2E 計測値）、`A` = 想定アバター遅延、`Lead` = 先行時間。

## 目的・ユースケース

**ローカル環境のユーザ（配信者）自身が、ソース音源を直接リアルタイムに聴きながら生演奏する**
（アバターを動かす）ケースに対応する。配信に乗るのはこのローカル生演奏であり、配信用チャンネルは
通常運用と同様に生でよい（録音済みである必要はない）。

### チャンネル構成（通常運用との違い）

- **通常運用**: 本プラグインを**単一チャンネル**に挿す。そのチャンネルから共演者向けブラウザ（WS）
  配信と OBS 配信の両方を行い、各出演者の聴取タイミングがそろうよう WS チャンネルディレイと
  配信ディレイを自動調整する。音源が生か録音済みかは問わない（録音済みを流し込むメリットは特にない）。
- **ローカル生演奏**: 生演奏するのはローカルのユーザ自身で、その聴取は**リアルタイム演奏が目的のため
  ディレイをかけられない＝調整できない**。そこで**共演者向けブラウザ（WS）配信専用の「先行チャンネル」を
  配信用チャンネルとは別に用意し、本プラグインはこの先行チャンネルに挿す**。
  - 先行チャンネルは配信より**先行**して出力する必要があり、その性質上**生配信を乗せられない**ため、
    やむを得ず**録音済みトラック**を流し込む（録音済みなのは先行のための都合）。

### 考え方：先行（マイナス遅延）を作り、必要分だけ相殺する

先行チャンネルへは、配信用チャンネルより `Lead` ms **先行して**録音済みトラックを流し込む。
これは実質的に、**あらかじめ大きめの「マイナス遅延（先行）」を作っておく**ことに相当する。

共演者は先行チャンネルを WS 経由で受け取るため、自身の WS 遅延・反応・アバター遅延があっても、
先行のぶんだけ"先取り"してローカル演奏者に追いつける。本機能は、この先行（マイナス遅延）を
**共演者向け WS チャンネルディレイの自動調整で必要分だけ相殺**し、調整できないローカル演奏者の
聴取タイミングへ共演者をそろえる。配信側に必要なオフセットは別途 UI が案内する（後述）。
これが本機能の主目的である。

先行（マイナス遅延）はあらかじめ十分大きく作っておくのが前提。不足すると相殺では足りず
「ディレイを減らす」必要が生じてしまい補正できない（→ 後述のエラー1）。

## タイミングモデル

ソース発音を起点とした相対時刻で考える（タイミング図の絶対時間軸に対応）。

- 共演者が音を聴くタイミング `min_lead = neg_max + R − A`（全チャンネルで同一）。
- ローカル演奏者が音を聴くタイミング = `Lead`（先行時間。物理的に固定で動かせない）。
- 両者を一致させるには、共演者側・配信側を `live_extra = Lead − min_lead` だけ後ろへずらす。
  - `live_extra ≥ 0` のときのみ成立（ディレイは増やせても減らせない）。

成立時:

```
各チャンネルディレイ total_ms += live_extra
配信ディレイ master_delay     = neg_max + live_extra = Lead − R + A
```

これで全レーンの右端（観客が知覚する時刻）が `Lead + A` にそろい、ローカルとリモートの
「出演者が聴くタイミング」(▼) が一致する。

## 2つの失敗条件

判定順は **エラー2 → エラー1**（エラー2はLeadで救えないため優先）。不成立時は `live_extra` を適用しない。

| 条件 | 内部フラグ | 意味 | 解決方法 |
|---|---|---|---|
| `R > A` | `live_service_too_slow` | 配信は生演奏（リアルタイム信号）を乗せるため、配信に要する追加オフセット `A − R` を負（＝前倒し）にできない。`R > A` で負になり先行時間では救えない | **より低遅延な配信サービス**（R を下げる）。UI は Error 表示 |
| `Lead < min_lead` | `live_lead_too_short` | 先行が足りずディレイを減らす必要が生じる（不可能） | 先行時間を `min_lead` ms 以上に。UI は最小値を Warning 表示 |

`R = A`（余裕ゼロ）は「負ではない」ため許容（縮退ケース）。

## 設計判断と実装意図

### 1. 本線(master_buf)へは自動ディレイを適用しない

生演奏成立時、`recalc_all_delays()` は `master_buf` のディレイを **0** にする
（[plugin-settings.cpp](../src/plugin/plugin-settings.cpp) `recalc_all_delays`）。

理由: 本プラグインを挿しているのは**先行チャンネル（共演者向けWS配信専用）であって配信本線ではない**。
本線(master_buf)にディレイをかけても配信タイミングは動かないため 0 とする。代わりに、配信側で必要な
オフセット（`A − R`）を **OBS ネイティブの音声同期オフセット**で手動設定してもらう。

UI では、配信チャンネルの親ソースへ設定すべき同期オフセット値を案内する:

```
sync_offset = REQUIRED_AUDIO_SYNC_OFFSET_MS(-950) + (A − R)
            = REQUIRED_AUDIO_SYNC_OFFSET_MS + (master_delay − Lead)
```

`-950 ms` は本プラグインが通常運用で親ソースに要求する基準同期オフセット
（[constants.hpp](../src/core/constants.hpp) `REQUIRED_AUDIO_SYNC_OFFSET_MS`）。案内は
全体調整グループ末尾の警告コールアウトと、プラグイングループ/WS計測タブ上部のバナーに表示する
（[properties-delay.cpp](../src/ui/properties-delay.cpp) / [properties-builder.cpp](../src/ui/properties-builder.cpp) `add_live_perf_offset_warning_if_needed`）。

> なお `master_delay_ms` スナップショット値自体は成立時 `neg_max + live_extra` を保持し、
> タイミング図と上記オフセット案内の計算に用いる（本線へ適用しないだけ）。

### 2. タイミング図は絶対時間軸へ切替

生演奏 ON 時、タイミング図は通常の「右端揃え（同期点）」から**ソース起点の絶対時間軸（左端揃え）**へ
切り替わる（[delay-diagram-widget.cpp](../src/widgets/delay-diagram-widget.cpp)）。

- 「Local」レーン `[先行時間(黒)] [アバター遅延]` を追加し、▼「出演者が聴くタイミング」を `Lead` に置く。
- 成立時は全レーンの▼・右端が一致。**不成立時はあえて揃えず**、ローカル▼が他とずれて見えることで
  問題を可視化する（`live_extra` 未適用のため）。
- 配信レーンは成立時のみ先行分を左に分割描画する（`build_broadcast_lane` の `live_split`）。

### 3. live_extra は計測済み/仮値チャンネルにのみ加算

全chが未計測のとき `total_ms` は 0 のままにすべきで、`live_extra` で汚さない
（[delay-state.hpp](../src/model/delay-state.hpp) `calc_all_delays`）。

## 運用上の注意

- 本プラグインは**共演者向け WS 配信専用の「先行チャンネル」に挿す**（配信本線とは別）。先行チャンネルへは
  配信用チャンネルより `Lead` ms 先行して**録音済みトラック**を給餌する（＝あらかじめマイナス遅延を作る）。
- 配信用チャンネル（生演奏が乗る本線）は通常運用と同様で可。ここへは本プラグインを挿さない。
- 配信側ディレイは自動適用されない。UI が案内する**同期オフセットを配信チャンネルの親ソースへ
  手動設定**する必要がある。
- 成立判定には `R`（RTSP計測）が必須。`min_lead` 算出には WS 計測値（`neg_max`）を用いる。
- `R` は `A` より小さいこと（`R > A` は配信サービスを変える以外に解決不能）。
