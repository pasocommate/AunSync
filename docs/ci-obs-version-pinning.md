# CI/ローカルビルドの OBS バージョン固定

## 方針

`OBS_STUDIO_REF` を `master` ではなく **OBS リリースタグ**に固定する。CI（`.github/workflows/build.yml`）とローカル（`build.bat` / `build.env`）が同じ値を参照するように単一定義にする。バージョンを更新する際はこの1か所だけ変更する。

**現在の固定値: `32.0.4`（サポート下限 OBS 32.0）**

`obs_module_ver` は 32.0 として焼かれ、patch 以下は OBS ローダが無視するため、32.0.x〜32.2 以降すべてで受理される（上方互換）。

## 根拠

- `master` を使うと OBS の内部 ABI 変化（libobs API やシンボル）が予告なく混入し、ロード互換が壊れる。
- 固定先を「サポートする最も古い OBS（下限）」にすると、古い OBS で拒否されず、新しい OBS でも `ver ≤ ランタイム` で受理される。
- サポート下限 = OBS 32.0 は、基準日 2026-06-26 時点で約9か月前のリリース。現行 32.2 から2マイナー前で保守負担が小さい。

## CI キャッシュの注意点

restore-keys のフォールバックを「同一 ref ＋同一 SHA」に限定する。広いフォールバックは古い OBS ビルドを再利用する温床になる。`OBS_STUDIO_REF` を変えたらキャッシュキーも変わり、`.deps` ごと取り直される設計にする。

## プラグインが使用する libobs API

プラグインが 32.0 以降で追加された libobs API を使わないこと。`32.0.4` のヘッダ・lib でビルドが通ることが事実上のガードになる。
